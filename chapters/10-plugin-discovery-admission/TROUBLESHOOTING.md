# 配套代码 10：常见报错

## 插件显示 `rejected`

先运行 `--demo-discovery` 查看静态原因。常见情况包括 manifest 字段缺失、入口文件不存在、声明的方法没有实现、使用禁止导入，或 read_only 工具却实现了副作用方法。

## 插件显示 `pending_confirmation`

这不是失败。插件声明自己会写文件或修改外部状态，因此自动检查通过后仍等待显式准入。真实系统应把确认与用户权限绑定；本章演示使用 `CONFIRM`。

## 插件显示 `stale`

准入记录中的指纹与当前 manifest / 源码不一致。不要手改 JSON 把状态改回 admitted。重新扫描、检查并准入当前版本。

## `probe 产生了文件副作用`

插件在本应只读的自检阶段创建或修改了文件。本章直接拒绝，即使 probe 自己返回 `ok: true`。修复插件，让 probe 只报告能力和依赖。

## `preview 产生了文件副作用`

预览不应真正执行动作。把写文件、发请求或修改状态的逻辑移到 execute，并让 preview 只返回计划摘要。

## `插件当前不可执行：unreviewed`

插件被发现，但还没有准入记录。先完成静态检查和准入；不要绕过 Runtime 直接导入插件。

## `插件当前不可执行：stale`

工具在准入后发生变化。重新准入前禁止执行，避免“批准 A、实际运行 B”。

## 每次都问很多问题

本章 Runtime 每轮只返回第一个缺失字段。如果上层界面一次显示了多个问题，说明它绕过了 `next_question()`，自行展开了 manifest。

## 相同计划仍写出第二份文件

检查插件名和所有输入是否完全相同。稳定计划键绑定完整输入；标题、目录或空格变化都会形成新计划。也要确认 `receipts.json` 没有被删除。

## 子进程超时或没有返回 JSON

插件的 probe / preview 应在短时间内结束，并返回可 JSON 序列化的对象。不要在这些阶段等待用户输入、启动长期服务或输出额外日志到 stdout。

## 这能安全运行网上下载的插件吗

不能。本章不是操作系统级沙箱。只对自己编写或已经人工审查的本地插件做实验；不要把未知来源代码放到真实账户、密钥和私人文件可访问的环境中运行。

## Windows 中文乱码

在 PowerShell 中执行：

```powershell
$env:PYTHONUTF8="1"
py main.py --self-test
```

GitHub Actions 工作流已经固定设置 `PYTHONUTF8=1`。
